Welcome. The central question for everyone here is not whether AI will change your job — it already has. The question is whether you design that change or it gets designed around you. This deck argues that the Iteration Manager and Delivery Manager are the natural locus for that design work, and it makes the case with evidence, not assertion. v2.1.0 opens on the stake, not the definition: the coordination gap is the hook.

Age of Agents

Agent capability is a ladder.

As agents climb it, your job moves from doing the work, to steering it, to governing it.

  • AssistedYou operate — the agent suggests; you do the work.
  • AugmentedYou review — the agent drafts and recommends; you execute and edit.
  • CollaborativeYou supervise — the agent plans and executes alongside you; you steer.
  • OrchestratedYou architect — agents hand off to each other; you set the gates and escalation.
  • AutonomousYou govern — agents run end-to-end; you audit and hold the kill-switch.

Synthesis · autonomy framing after Feng, McDonald & Zhang (Univ. of Washington, 2025); cf. SAE J3016, Cloud Security Alliance 2026

05
aha agile
This ladder is a synthesis of industry agentic-maturity models (Microsoft, Salesforce, Adaline, Bessemer), independently grounded in the University of Washington "Levels of Autonomy for AI Agents" paper (Feng, McDonald and Zhang, 2025), the Cloud Security Alliance autonomy levels (2026), and the SAE J3016 analogy. The load-bearing point: as agents mature, the human's job shifts from doing the work, to steering it, to governing it — gatekeeping becomes auditing. Keep it distinct from the AI Governance Maturity ladder (now in the appendix), which measures how well an organisation governs agents, not what the agents can do. Source confidence: High for the autonomy-spectrum concept (triangulated across four independent sources); the five-rung labelling is a presentation synthesis, not a single citable framework.
The Loop

The loop runs until it hits a gate you designed.

06
aha agile
The loop is what makes an agent feel different from a chatbot. A chatbot stops when you stop typing. An agent doesn't. It receives a trigger — a scheduled time, an incoming ticket, a threshold crossed. It thinks, acts, observes the result, then decides: am I done, or another pass. It loops until the task is complete or it hits a gate you've designed. That gate — the human-in-the-loop decision point — is exactly where the IM's governance role sits. The loop doesn't stop itself. Someone decides when human judgment enters the cycle. (The chat-vs-agent anatomy is in the appendix if the room needs the "what is an agent" primer.)
From the Trenches
The first boundary I owned
was one nobody was watching.
A team I ran let an agent draft and publish every release note for a sprint. Fast, tidy, no complaints. Then one went out with a customer's name in it. The agent hadn't decided to include it. Nobody had decided what it was allowed to publish. The fix wasn't more proofreading. It was naming who signs off before an agent ships. That's what owning the boundary looks like.

The Mechanism

You decide which pattern. The agent can't.

This is workflow design — set before the agent runs. That's architecture, not the in-task judgment calls where adding AI degrades the decision.

Code, docs, plans

Human steers → AI executes
Complex analysis, research
Your move: maintain the backlog the agents pull from.
AI monitors → Human intervenes
Ops, alerts, QA
Your move: set the escalation thresholds.
16
aha agile
The HITL taxonomy comes from LangGraph documentation, the DEV Community HITL taxonomy, and Anthropic's Trustworthy Agents guidelines — engineering documentation, not controlled studies of IM behavior. The mapping from each pattern to an IM accountability decision is argued from functional alignment: each pattern requires a governance decision natively in the IM's space, not the tech lead's. The tech lead designs the agent. The IM decides which human-in-the-loop pattern governs how that agent participates in the delivery system. The three IM-role entries — DoD gate design, task backlog stewardship, escalation threshold setting — are the concrete G-NEW-1 connection. The Malone distinction matters for analytical audiences: the IM's role is selecting which pattern applies to which work (a design decision made before the work begins), not making decisions augmented by AI within a task.

Evolution path

This fork is real. The argument for evolution is cost: one accountable owner beats handoffs between separate specialists.
17
This is the crisis of the deck — evolve or dissolve. Use it for the "the role will just dissolve anyway" objection. Fragmentation is real and will happen in some organisations. The argument for evolution is probabilistic — not every IM will make the governance decision in time. The structural case is handoff cost: splitting workflow architecture and human-AI risk across a tech lead and a separate governance specialist creates new coordination seams where today there is one owner. Organisations that fragment the role tend to rediscover this within a couple of years as coordination overhead returns in a new form. Note: prioritisation is deliberately not shown as a fragmentation outcome — that is legitimately the product manager's job, not a piece of the IM role being taken away.

Appendix — What Is an Agent

Chat ends when you stop typing. Agents don't.

26
aha agile
Use this if the room needs the "what is an agent" primer. Chat is stateless: you ask, the model responds, done — no state, no tools, no follow-through. An agent is different in kind. The LLM is still the reasoning engine, but it now has four things chat doesn't: Context (persistent memory and a knowledge base it can query), Connections (tools and APIs it can invoke), Capabilities (multi-step skills it executes without re-prompting), and Cadence (it can be triggered on a schedule or event, not just by a human typing). That shift from "model that responds" to "system that runs" is what agentic workflow means.
