The Shift 04

Chat ends when you stop typing. Agents don't.

Hybrid Delivery Adnan Ali · 2026
Start here every time you use the word "agent" — because most people in the room are thinking "chatbot with a longer context window." They're not wrong. Chat is exactly that: you ask, the model responds, done. No state, no tools, no follow-through. An agent is different in kind. The LLM is still the reasoning engine — the brain — but now it has four things around it that chat doesn't: Context (persistent memory and a knowledge base it can query), Connections (tools and APIs it can invoke — search, calendar, code execution), Capabilities (multi-step skills it can execute without you re-prompting), and Cadence (it can be triggered on a schedule or by an event, not just by a human typing). That shift from "model that responds" to "system that runs" is what agentic workflow means. The next slide shows what that looks like in motion.
The Loop 05

The loop runs until it hits a gate you designed.

Hybrid Delivery Adnan Ali · 2026
The loop is what makes an agent feel different from a chatbot. A chatbot stops when you stop typing. An agent doesn't. It receives a trigger — a scheduled time, an incoming ticket, a threshold crossed in a dataset. It thinks: what do I know, what do I need to do. It acts: searches, queries, writes, updates. It observes the result: did that work, what changed. Then it decides: am I done, or do I need another pass. It loops until the task is complete or it hits a gate you've designed. That gate — the human-in-the-loop decision point — is exactly where the IM's governance role sits. The loop doesn't stop itself. Someone decides when human judgment enters the cycle.

Hybrid Delivery Adnan Ali · 2026
The HITL taxonomy comes from LangGraph documentation, the DEV Community HITL taxonomy, and Anthropic's Trustworthy Agents guidelines. These are engineering documentation sources, not controlled studies of IM behavior. The mapping from each pattern to a specific IM accountability decision is argued from functional alignment: each pattern requires a governance decision that is natively in the IM's accountability space, not the tech lead's. The tech lead designs the agent. The IM decides which human-in-the-loop pattern governs how that agent participates in the team's delivery system. The three IM-role entries — DoD gate design, task backlog stewardship, escalation threshold setting — are the concrete G-NEW-1 connection: these are IM-specific decisions, not engineering decisions. The Malone distinction is critical for analytically-minded audiences: Malone et al. (Slide 6) establishes that adding AI to decision tasks degrades performance below the best solo performer. This is not a contradiction of the HITL argument — the IM's governance role is selecting which pattern applies to which work (a workflow design decision made before the work begins), not making decisions augmented by AI within a task. The distinction matters: an IM deciding "this workflow uses AI Monitors → Human Intervenes" is doing workflow architecture, not AI-augmented decision-making.

Use this slide for the "the role will just dissolve anyway" objection. Fragmentation is real and will happen in some organisations. The argument for evolution is probabilistic — not every IM will make the governance decision in time. The structural case for evolution is handoff cost: splitting workflow architecture and human-AI risk across a tech lead and a separate governance specialist creates new coordination seams where today there is one owner.
